class DataSourceScheduler:
    """数据源调度中心"""
    
    def __init__(self):
        self.sources = {}  # 数据源注册表
        self.priorities = {}  # 优先级配置
        self.health = {}  # 健康度追踪
    
    def register_source(self, name, fetch_func, priority):
        """注册数据源"""
        pass
    
    def fetch_with_fallback(self, field, code):
        """自动降级获取"""
        for source in self.priorities[field]:
            data = source.fetch(code)
            if data:
                return data
        return None
```

**优势**:
- 统一管理所有数据源

class DynamicWeightScorer:
    """动态权重评分"""
    
    def calculate_dynamic_weights(self, market_condition):
        """根据市场情况动态调整权重"""
        if market_condition == 'bull':
            # 牛市：降低股息率权重，提高波动率权重
            return {'dividend_yield': 0.4, 'volatility': 0.4, 'stability': 0.2}
        elif market_condition == 'bear':
            # 熊市：提高股息率权重，降低波动率权重
            return {'dividend_yield': 0.6, 'volatility': 0.2, 'stability': 0.2}
        else:
            # 震荡市：均衡配置
            return {'dividend_yield': 0.5, 'volatility': 0.3, 'stability': 0.2}
```

**优势**:
- 适应市场变化
- 提高策略有效性

### 3.2 核心经验教训

#### 经验1: 数据接口是最大瓶颈

**教训**:
```
外部接口不可控：
- 频率限制
- 稳定性差
- 数据质量参差不齐
- 可能随时失效
```

**经验**:
```python
# ✅ 正确做法
多数据源架构:
  主数据源 → 备数据源 → 计算方式 → 优雅降级

监控告警:
  实时监控数据源健康度
  异常立即告警

文档化:
  记录所有接口的特性、限制、稳定性
```

**应用**:
- v6.15实现价格双重验证
- v6.18实现健康度追踪器
- 建立了完整的数据源文档

---

#### 经验2: 测试驱动开发是必需的

**教训**:
```
v6.15第一轮测试：成功
v6.15第二轮测试：接口限制，全部失败

没有测试，问题会在生产环境爆发
```

**经验**:
```
测试流程:
1. 单元测试（核心函数）
2. 集成测试（完整流程）
3. 性能测试（大数据量）
4. 回归测试（修改后重新测试）
```

**应用**:
- v6.18创建了test_v18_basic.py
- 100%测试通过
- 建立了测试脚本库

---

#### 经验3: 简单设计优于复杂架构

**教训**:
```
过度设计:
- 复杂的状态机
- 多层抽象
- 过早优化

结果:
- 开发周期长
- 维护成本高
- 容易出bug
```

**经验**:
```
极简主义:
1. 最小可行方案
2. 不过度设计
3. 优先解决最痛的问题
4. 迭代改进
```

**应用**:

### 3.3 开发模式总结

#### 模式1: 第一性原理

**核心**: 回归问题本质，找到最痛的点

**案例**:
```
问题: 单源模式可信度标记为"低"
本质: 缺乏数据源质量评估机制
方案: 健康度追踪器（v6.18）

问题: 验证需要25分钟
本质: 串行请求效率低
方案: 并发优化（v6.18）
```

**应用**:
- 每次改进都先问：核心问题是什么？
- 不被表象迷惑，直击本质

---

#### 模式2: 极简主义

**核心**: 最小可行方案，聚焦核心问题

**案例**:
```
健康度追踪器:
  只追踪3个指标：成功率、延迟、稳定性
  简单计算：加权平均
  不做复杂的预测模型

缓存机制:
  内存字典，不用Redis
  简单TTL，不用LRU
  最小可行
```

**应用**:
- 每个功能都问：最简单的方案是什么？
- 避免"大而全"，追求"小而美"

---

#### 模式3: 快速迭代

**核心**: 小步快跑，快速验证

**案例**:
```
v6.0-v6.13: 13个版本，快速迭代
v6.14-v6.18: 5个版本，持续优化

每个版本:
- 1-3个核心改进
- 完整测试验证
- 文档记录
```

**应用**:
- 不追求一步到位
- 每次改进一点点
- 快速验证，快速调整

---

#### 模式4: 文档驱动

**核心**: 先规划，后执行

**案例**:
```
v6.18改进流程:
1. 先更新PRD，明确需求
2. 再写设计思路（VALIDATION_LIMITATIONS_AND_IMPROVEMENTS.md）
3. 然后实施代码
4. 最后写总结报告（V18_IMPROVEMENT_REPORT.md）
```

**应用**:
- 文档是思维的载体
- 好记性不如烂笔头
- 文档让思路更清晰

---

#### 模式5: 持续反思

**核心**: 每次改进后总结经验

**案例**:

**应用**:
