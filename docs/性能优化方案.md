# 性能优化方案 - 数据缓存与快速重筛

## 问题分析

**当前问题**：
- 每次修改配置后需要重新运行系统（3-5分钟）
- 大部分数据与配置无关，重复计算是浪费
- 用户体验差，等待时间长

## 数据分类

### 1. 静态数据（与配置无关）
这些数据不随配置变化，每天只需获取一次：

**基础数据**：
- 价格、市值、PE、PB
- 股息率（TTM）、波动率
- ROE、负债率、EPS
- 分红年数、股利支付率

**质量因子数据**：
- 利润增速（近3年CAGR）
- ROE趋势、PEG
- 现金流质量、股东持股比例

**技术指标数据**：
- 均线位置（MA250、MA20、MA60）
- 趋势、信号级别
- 均线斜率、价格相对位置

**获取时间**：3-5分钟
**更新频率**：每日一次

### 2. 动态数据（依赖配置）
这些数据会随配置变化：

**筛选结果**：
- 哪些股票入选（根据MIN_DIVIDEND_YIELD、MIN_MARKET_CAP等）
- 排除哪些股票（根据ROE、负债率、分红年数等）

**评分数据**：
- 综合评分（权重可变：WEIGHT_DIVIDEND、WEIGHT_VOL等）
- 排名（根据评分排序）

**计算时间**：<5秒
**更新频率**：配置变化时

## 优化方案

### 架构设计

#### 方案A：双表分离（推荐）

```
stock_raw_data（静态数据表）
├── 基础数据：价格、市值、财务指标
├── 质量因子：利润增速、ROE趋势
├── 技术指标：均线、趋势、信号
└── 更新时间：每日一次

stock_data（筛选结果表）
├── 筛选结果：哪些股票入选
├── 评分数据：综合评分、排名
└── 更新时间：配置变化时
```

**优势**：
- 静态数据复用，避免重复获取
- 快速重筛，配置变化后几秒完成
- 数据分离，逻辑清晰

#### 方案B：单表优化

```
stock_data（单表）
├── 所有数据（静态+动态）
├── 添加标记：is_screened（是否已筛选）
└── 配置变化时只重新筛选和评分
```

**优势**：
- 改动较小，易于实现
- 不需要修改数据库结构

**劣势**：
- 每次运行都清空表，无法区分静态和动态数据
- 需要添加复杂逻辑判断

### 2. API设计

#### 2.1 `/api/fetch` - 获取静态数据

**功能**：只获取静态数据，不筛选
**耗时**：3-5分钟
**调用时机**：
- 每日首次运行
- 用户点击"刷新数据"按钮

**请求**：
```json
POST /api/fetch
{
  "force": false  // 是否强制刷新
}
```

**响应**：
```json
{
  "success": true,
  "count": 5000,
  "data_date": "2026-04-01",
  "fetch_time": 180.5,
  "message": "数据获取成功（耗时180.5秒）"
}
```

#### 2.2 `/api/rescreen` - 快速重筛

**功能**：基于已有静态数据进行筛选和评分
**耗时**：<5秒
**调用时机**：
- 配置变化后自动调用
- 用户点击"重新筛选"按钮

**请求**：
```json
POST /api/rescreen
{
  "strategy": "aggressive"  // 可选，指定策略
}
```

**响应**：
```json
{
  "success": true,
  "count": 81,
  "data_date": "2026-04-01",
  "screen_time": 2.3,
  "message": "筛选完成（耗时2.3秒）"
}
```

#### 2.3 `/api/run` - 完整运行（保留）

**功能**：获取数据 + 筛选（用于首次运行）
**耗时**：3-5分钟
**调用时机**：
- 首次运行
- 用户点击"完整运行"按钮

## 技术细节

### 数据一致性保证

1. **静态数据版本控制**：
   - 每次获取静态数据时记录日期
   - 筛选时检查静态数据日期是否为今天
   - 如果过期，提示用户刷新数据

2. **筛选结果标记**：
   - 记录使用的配置参数
   - 记录筛选时间
   - 记录筛选耗时

### 错误处理

1. **静态数据不存在**：
   - 自动调用`/api/fetch`获取数据
   - 提示用户等待数据获取完成

2. **静态数据过期**：
   - 提示用户数据已过期
   - 提供"刷新数据"按钮

3. **筛选失败**：
   - 回退到上次成功的筛选结果
   - 显示错误信息

---

## 总结

这个优化方案将配置切换的响应时间从3-5分钟降低到<5秒，提升60-100倍。用户可以快速尝试不同配置和策略，大幅提升使用体验。

**核心思想**：将不变的数据（静态数据）和可变的数据（筛选结果）分离，避免重复计算。
